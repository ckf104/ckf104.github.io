### Execution, Memory Dependency and Sync Scope
vulkan 中的同步操作围绕 *sync scope* 和 *access scope* 这两个概念展开

*sync scope* 由 src sync scope 和 dst sync scope 组成，它指定 dst sync scope 中的操作在 src sync scope 完成之后才能开始，这对应一个 *execution dependency*。如果前一个 *execution dependency* 的 dst sync scope 和后一个 *execution dependency* 的 src sync scope 交集非空，那么它们可以串起来形成一个 *execution dependency chain*。容易推导出，**在一个 *execution dependency chain* 上，任意一个靠后的 dst sync scope 一定在任意一个靠前的 src sync scope 执行完成后才开始执行** 

但是只有 *execution dependency* 是不够的，因为 gpu 上有许多种类的 cache（访问 buffer 和访问 texture 可能就用的不同的 cache），这些 cache 之间可能不具备 coherency，因此还需要指定前后操作的 *memory dependency* 来正确地同步 memory 以避免 WAW 和 RAW 冒险（虽然 WAR 冒险只需要 *execution dependency* 就可以避免了）

[Execution and Memory Dependencies](https://registry.khronos.org/vulkan/specs/latest/html/vkspec.html#synchronization-dependencies) 中定义了三个基本的涉及 memory 同步的操作
* _Availability operations_ cause the values generated by specified memory write accesses to become _available_ to a memory domain for future access
* _Memory domain operations_ cause writes that are available to a source memory domain to become available to a destination memory domain (an example of this is making writes available to the host domain available to the device domain)
* _Visibility operations_ cause values available to a memory domain to become _visible_ to specified memory accesses

这里的定义是直接引用的原文。memory domain 的大概意思就是像 device memory 和 host memory 之类的。然后 *availability operations* 是指在同一个 memory domain 中，之前写的数据变为 available（我理解大概率在硬件上对应一个 flush cache 到主存的操作）。然后 _memory domain operations_ 是把在一个 memory domain 下已经 available 的数据让另一个 memory domain 也 available（类似于把 gpu 共享的 L2 cache flush 到主存），最后 _visibility operations_ 是让后续的读写能够 visible 已经 available 的数据（我理解大概率在硬件上对应一个 invalid cache 的操作）

**注：_visibility operations_ 是让 dst sync scope 中指定类型的读写能看到已经 avaiable 的数据，并不局限于本 denpendency 的 *availability operations* 产生 available 的数据，因此这里的 _visibility operations_ 和 *availability operations* 是完全解耦的**

access scope 就指定了在 src sync scope 中哪些写操作需要 *availability operations*，然后 dst sync scope 中哪些读写操作需要 _visibility operations_ ，这样保证 dst sync scope 的读写操作能看到 src sync scope 的读写操作。这里是有一个严格的序关系，**因此保证了在 *execution dependency chain* 中任意一个靠后的 dst sync scope 的 access scope 中的读写操作能够看到任意一个靠前的 src sync scope 的 access scope 中的写操作**（我之前一直疑惑为什么有时候 src / dst access scope 会设置为空，这里就解释了，因为这时我可能只需要 flush cache 或者 invalid cache 之类的）

另外，当 src access scope 在包含 `VK_ACCESS_HOST_WRITE_BIT` 时，会包含一个 _memory domain operations_，使得在 host domain available 的写操作也会在 device domain available。类似地，当 dst access scope 包含 `VK_ACCESS_HOST_READ_BIT` 或者 `VK_ACCESS_HOST_WRITE_BIT` 时，它也会包含一个 _memory domain operations_，使得在 device domain available 的写操作也将在 host domain available

**注：[VkBufferMemoryBarrier](https://registry.khronos.org/vulkan/specs/latest/man/html/VkBufferMemoryBarrier.html) 和 [VkImageMemoryBarrier](https://registry.khronos.org/vulkan/specs/latest/man/html/VkImageMemoryBarrier.html) 对 `VK_ACCESS_HOST_WRITE_BIT` 和 `VK_ACCESS_HOST_READ_BIT` 的描述有些微妙的差异，我暂时不是很理解这个地方**

[vkInvalidateMappedMemoryRanges](https://registry.khronos.org/vulkan/specs/latest/man/html/vkInvalidateMappedMemoryRanges.html)中举了一个特别好的例子，host 看到 device 对 memory 的写需要经历下面的流程
* Device write
* Device memory barrier including host reads in its second scope（*availability operations* 和 _memory domain operations_）
* Signal on the device (e.g. a [fence signal operation](https://registry.khronos.org/vulkan/specs/latest/html/vkspec.html#synchronization-fences-signaling))
* Wait on the host (e.g. [vkWaitForFences](https://registry.khronos.org/vulkan/specs/latest/html/vkspec.html#vkWaitForFences))
* [vkInvalidateMappedMemoryRanges](https://registry.khronos.org/vulkan/specs/latest/html/vkspec.html#vkInvalidateMappedMemoryRanges)（_visibility operations_，可能会刷 CPU 这边的 cache）

总结一下，如果是希望在 memory domain A 上的写能被 memory domain B 上的读写操作看见的话，需要
* *availability operations*：指定包含该写操作的 src access cope，使得该写操作在 domain A 上 available
* _memory domain operations_：使得该写操作在 domain B 上 avaiable（如果 A 和 B 是同一个 domain 就不需要这一步了）
* _visibility operations_：使得该写操作对后续的指定 access cope 的 domain B 上的读写操作可见

讨论完 *sync scope* 和 *access scope*，我们来看看 vulkan 的同步原语是如何指定 *sync scope* 和 *access scope* 的
### Fence
fence 用于 device 和 host 的同步，[synchronization-fences](https://registry.khronos.org/vulkan/specs/latest/html/vkspec.html#synchronization-fences) 文档中对在 queue submission command 中使用 fence 时，它的 *sync scope* 和 *access scope* 有更准确的描述。不过粗略地来讲，它保证在 fence signal operation 之后，在 queue submission 中提交的 commands，以及更早之前提交的 commands，它们的写操作在 device domain 上都 available 了
### Binary and Timeline Semaphores
binary semaphore 只有 reset 和 signal 两种状态。刚创建出来是 reset 状态，然后一些函数会产生 semaphore signal operation，将其设置为 signal 状态，通过 wait semaphore 可以将 signal 状态的 semaphore 又设置为 reset 状态（然后又可以复用该 semaphore），一个相关的讨论见 [Can I check if a semaphore has signaled already?](https://community.khronos.org/t/can-i-check-if-a-semaphore-has-signaled-already/108055)

但 binary semaphore 有许多局限，例如在 [vkQueueSubmit](https://registry.khronos.org/vulkan/specs/latest/man/html/vkQueueSubmit.html) 中要求
> All elements of the `pWaitSemaphores` member of all elements of `pSubmits` created with a [VkSemaphoreType](https://registry.khronos.org/vulkan/specs/latest/man/html/VkSemaphoreType.html) of `VK_SEMAPHORE_TYPE_BINARY` **must** reference a semaphore signal operation that has been submitted for execution and any [semaphore signal operations](https://registry.khronos.org/vulkan/specs/latest/html/vkspec.html#synchronization-semaphores-signaling) on which it depends **must** have also been submitted for execution

即需要先 submit signal command 再 submit wait command，也不是很理解为啥会有这样的顺序要求

另一方面，vulkan 1.2 引入了 Timeline Semaphores 则是用 64 bits 数来表示状态，要求其状态单调递增。wait 时指定一个数值，当 semaphore 的数值大于等于 wait 的值时 wait 结束，并且允许 wait before signal 操作

**但同时 [Vulkan Timeline Semaphores](https://www.khronos.org/blog/vulkan-timeline-semaphores) 中谈到 WSI API 尚不支持 timeline semaphore**，在 vulkan 实践一节中我们会谈到这带来的一些影响

[Semaphore Signaling](https://registry.khronos.org/vulkan/specs/latest/html/vkspec.html#synchronization-semaphores-signaling) 和 [Semaphore Waiting](https://registry.khronos.org/vulkan/specs/latest/html/vkspec.html#synchronization-semaphores-waiting) 定义了 queue submission command 中使用 semaphore 时的 *sync scope* 和 *access scope*。不过粗略来讲，semaphore signal operation 使得在 queue submission 中提交的 commands，以及更早之前提交的 commands，它们的写操作在 device domain 上都 available 了。而 semaphore waiting operation 使得在 waiting queue submission 中的 commands，以及后续的 commands 都能看见 semaphore signal operation 保证的 available 的写操作了
### Pipeline Barrier
`vkCmdPipelineBarrier2` 的定义的 barrier 是更细粒度的同步操作，它可以由用户指定具体的 sync scope 和 access scope

src sync scope 由在 _submission order_ 在该 command 之前的 command 组成，dst sync scope 由在 _submission order_ 在该 command 之后的 command 组成。`VkPipelineStageFlags2` 可以指定这些 commands 具体要同步的 pipeline stage。对于 `srcStageMask` 而言，`VkPipelineStageFlags2` 包含的 pipeline stage，以及隐式地在该 pipeline stage 之前的 pipeline stage，都是在 src sync scope 中的。而对于 `dstStageMask` 而言，`VkPipelineStageFlags2` 包含的 pipeline stage，以及隐式地在该 pipeline stage 之后的 pipeline stage，都是在 dst sync scope 中的

**注：根据 [pipeline-barriers](https://registry.khronos.org/vulkan/specs/latest/html/vkspec.html#synchronization-pipeline-barriers) 文档，如果 barrier commands 位于一个 subpass 内，那么 src sync scope 和 dst sync scope 都仅局限在该 subpass 内**

global / buffer / image memory barrier 进一步指定了 barrier 的 access scope。src access scope 由 src sync scope 中的 commands 的由 `srcStageMask` 指定的 pipeline stage 中的，并且是由 `srcAccessMask` 指定的 memory access type 的，并且 memory barrier 指定的内存范围的 memory access 组成。dst access scope 由 dst sync scope 中的 commands 的由 `dstStageMask` 指定的 pipeline stage 中的，并且是由 `dstAccessMask` 指定的 memory access type 的，并且 memory barrier 指定的内存范围的 memory access 组成

其中 buffer / image barrier 分别指定了 buffer 和 image 中需要执行 available 和 visible 操作的内存范围，而 global memory barrier 不限制内存范围

另外，barrier 中可能包含 [Image Layout Transitions](https://registry.khronos.org/vulkan/specs/latest/html/vkspec.html#synchronization-image-layout-transitions) 操作，**因为 layout transition 可能会对 image memory 进行读写，标准规定了 layout transition 发生的时机在 _available operation_  之后， _visibility operation_ 之前**
> Image subresources **can** be transitioned from one [layout](https://registry.khronos.org/vulkan/specs/latest/html/vkspec.html#resources-image-layouts) to another as part of a [memory dependency](https://registry.khronos.org/vulkan/specs/latest/html/vkspec.html#synchronization-dependencies-memory) (e.g. by using an [image memory barrier](https://registry.khronos.org/vulkan/specs/latest/html/vkspec.html#synchronization-image-memory-barriers)). When a layout transition is specified in a memory dependency, it happens-after the availability operations in the memory dependency, and happens-before the visibility operations. Image layout transitions **may** perform read and write accesses on all memory bound to the image subresource range, so applications **must** ensure that all memory writes have been made [available](https://registry.khronos.org/vulkan/specs/latest/html/vkspec.html#synchronization-dependencies-available-and-visible) before a layout transition is executed. Available memory is automatically made visible to a layout transition, and writes performed by a layout transition are automatically made available.

#### TOP_OF_PIPE / BOTTOM_OF_PIPE
在 vulkan 1.0 的时候引入了 `VK_PIPELINE_STAGE_TOP_OF_PIPE_BIT` 和 `VK_PIPELINE_STAGE_BOTTOM_OF_PIPE_BIT` 这俩特殊的 pipeline stage，前者表示任意一个 command 开始的 stage，因此当 `VK_PIPELINE_STAGE_TOP_OF_PIPE_BIT` 用于 `srcStageMask` 时就表示 src sync scope 为空，而用于 `dstStageMask` 时则表示 barrier 后边的所有指令。`VK_PIPELINE_STAGE_BOTTOM_OF_PIPE_BIT` 则恰好相反

但在指定 access scope 时会遇到一些问题，虽然 sync scope 中 pipeline stage 具有延展性（即 src sync scope 包含了某个 pipeline stage，那也会隐式地包含更早的 pipeline stage），但 **access scope 是不具有这样的延展性的**。那用户现在在 `dstStageMask` 中使用 `VK_PIPELINE_STAGE_TOP_OF_PIPE_BIT` 表示 sync scope 要包含后续所有的指令时，access scope 就没法指定了，因为 `VK_PIPELINE_STAGE_TOP_OF_PIPE_BIT` 不包含任何的 pipeline stage，也就没有任何的 memory access 了。**这是因为 access scope 没有延展性，因此它只会包括 stage mask 中指定的 pipeline stage 中的 memory access（即使后续的 pipeline stage 隐式地包含在 sync scope 里）**

类似讨论可见 [VK_PIPELINE_STAGE_TOP_OF_PIPE_BIT VkAccessFlags set to 0?](https://stackoverflow.com/questions/65764113/vk-pipeline-stage-top-of-pipe-bit-vkaccessflags-set-to-0) 以及 [Yet another blog explaining Vulkan synchronization](https://themaister.net/blog/2019/08/14/yet-another-blog-explaining-vulkan-synchronization/)
虽然但是，那我写成 `dstStageMask = VK_PIPELINE_STAGE_TOP_OF_PIPE_BIT | OTHER_STAGE_BITS` 之类的，把其它包含 memory access 的 stage 也加上不就行了吗。不过我没找到类似的讨论

无论如何，在 vulkan 1.3 中，引入了 `VK_PIPELINE_STAGE_2_NONE`。比起 `VK_PIPELINE_STAGE_2_TOP_OF_PIPE_BIT` 和 `VK_PIPELINE_STAGE_2_BOTTOM_OF_PIPE_BIT`，使用 `VK_PIPELINE_STAGE_2_NONE` 和 `VK_PIPELINE_STAGE_2_ALL_COMMANDS_BIT`（包含所有的 pipeline stage，因此避免了延展性的问题）是更推荐的选择
### Event
event 又被称为 split barrier，它将 pipeline barrier 细分为了 signal 和 wait 两个阶段，因此能做到更细粒度的同步

vulkan 1.3 推荐 `vkCmdSetEvent2` 和 `vkCmdWaitEvents2` 搭配使用，前者指定 event signal operation，后者指定 event wait operation，`pDependencyInfo` 用于指定同步的 sync scope 和 access scope，并且 src sync scope 限制在 `vkCmdSetEvent2` 之前的指令，而 dst sync scope 限制在 `vkCmdWaitEvents2` 之后的指令（因此称为 split barrier）。[vkCmdWaitEvents2](https://registry.khronos.org/vulkan/specs/latest/man/html/vkCmdWaitEvents2.html) 要求这俩 command 使用的 `pDependencyInfo` 是相同的
> The event was signaled by a [vkCmdSetEvent2](https://registry.khronos.org/vulkan/specs/latest/man/html/vkCmdSetEvent2.html) command that occurred earlier in [submission order](https://registry.khronos.org/vulkan/specs/latest/html/vkspec.html#synchronization-submission-order) with a `dependencyInfo` parameter exactly equal to the element of `pDependencyInfos` at index i

而在 vulkan 1.0 中是 `vkCmdSetEvent` 和 `vkCmdWaitEvents` 搭配使用。这俩 API 最大的问题是 `vkCmdSetEvent` 只有一个 `stageMask` 参数来指定 src sync scope，而 src access cope 以及 dst  sync scope 和 image layout transition 等信息得等到 `vkCmdWaitEvents` 中才指定：
> Since [vkCmdSetEvent](https://registry.khronos.org/vulkan/specs/latest/man/html/vkCmdSetEvent.html) does not have any dependency information beyond a stage mask, implementations do not have the same opportunity to perform [availability and visibility operations](https://registry.khronos.org/vulkan/specs/latest/html/vkspec.html#synchronization-dependencies-available-and-visible) or [image layout transitions](https://registry.khronos.org/vulkan/specs/latest/html/vkspec.html#synchronization-image-layout-transitions) in advance as they do with [vkCmdSetEvent2](https://registry.khronos.org/vulkan/specs/latest/man/html/vkCmdSetEvent2.html) and [vkCmdWaitEvents2](https://registry.khronos.org/vulkan/specs/latest/man/html/vkCmdWaitEvents2.html).

因为 `srcStageMask` 也前后出现了两次，[vkCmdWaitEvents](https://registry.khronos.org/vulkan/specs/latest/man/html/vkCmdWaitEvents.html) 要求：
>`srcStageMask` **must** be the bitwise OR of the `stageMask` parameter used in previous calls to `vkCmdSetEvent` with any of the elements of `pEvents` and `VK_PIPELINE_STAGE_HOST_BIT` if any of the elements of `pEvents` was set using `vkSetEvent`
### Framebuffer Region Dependencies
我感觉 [Framebuffer Region Dependencies](https://registry.khronos.org/vulkan/specs/latest/html/vkspec.html#synchronization-framebuffer-regions) 纯粹是为了 tile-based architectures 打的补丁，因为按照之前的 *execution dependency* 和 *memory dependency* 的定义，如果 subpass B 依赖 subpass A 的输出，那么需要 subpass A 整个执行完了才能开始执行 subpass B，这肯定不是我们希望的（因为片上的 tile memory 肯定装不下 subpass A 的全部输出）

因此 [Framebuffer Region Dependencies](https://registry.khronos.org/vulkan/specs/latest/html/vkspec.html#synchronization-framebuffer-regions) 将 *memory dependency* 进一步分为三类
* 如果 dependency 的 pipeline stage 中含有 `VK_PIPELINE_STAGE_FRAGMENT_SHADER_BIT`，`VK_PIPELINE_STAGE_EARLY_FRAGMENT_TESTS_BIT`，`VK_PIPELINE_STAGE_LATE_FRAGMENT_TESTS_BIT`，`VK_PIPELINE_STAGE_COLOR_ATTACHMENT_OUTPUT_BIT` 这四个之外的 stage，那么就是普通的 *memory dependency*
* 否则，并且 `dependencyFlags` 参数中不包含 `VK_DEPENDENCY_BY_REGION_BIT`，它是一个 *framebuffer-global dependency*
* 否则，它是一个 *framebuffer-local dependency*

其中 `dependencyFlags` 参数在 `vkCmdPipelineBarrier2` 以及 `VkSubpassDependency` 中都有

*framebuffer-local dependency* 是我们需要的，**它的 sync scope 仅包含同一个 *framebuffer region* 内的读写操作**（不同的 *framebuffer region* 大小在文档中进一步分类为 _sample region_，_fragment region_ 和 *tile region*）

例如对于 _sample region_ 来讲，这个同步粒度就是前面一个 subpass 对 framebuffer 中的一个采样点的写，会与后面一个 subpass 对 framebuffer 中同一个采样点的读进行同步，而不同 subpass 对 framebuffer 中不同位置的读写顺序是没有保证的

### Sync in Render Pass
[VkSubpassDependency](https://registry.khronos.org/vulkan/specs/latest/man/html/VkSubpassDependency.html) 告诉我们，每个 `VkSubpassDependency` 都对应一个同步操作
* src sync scope 由 `srcSubpass` 指定，如果 `srcSubpass` 等于 `VK_SUBPASS_EXTERNAL` 的话，src sync scope 表示 `vkCmdBeginRenderPass` 之前的 commands 中的由 `srcStageMask` 指定的 pipeline stage
* dst sync scope 由 `dstSubpass` 指定，如果 `dstSubpass` 等于 `VK_SUBPASS_EXTERNAL` 的话，dst sync scope 表示 `vkCmdEndRenderPass` 之后的 commands 中的由 `dstStageMask` 指定的 pipeline stage
* src access scope 和 dst sync scope 由 `srcAccessMask` 和 `dstAccessMask`

当然，这个同步操作还会包含 dst subpass 依赖的 attachments 的 layout 转换

另外，对任意一个 attachment，如果 `VK_SUBPASS_EXTERNAL` 到第一个使用该 attachment 的 subpass 之间没有 subpass denpendency，那么会隐式地插入一个
```
VkSubpassDependency implicitDependency = {
    .srcSubpass = VK_SUBPASS_EXTERNAL,
    .dstSubpass = firstSubpass, // First subpass attachment is used in
    .srcStageMask = VK_PIPELINE_STAGE_NONE,
    .dstStageMask = VK_PIPELINE_STAGE_ALL_COMMANDS_BIT,
    .srcAccessMask = 0,
    .dstAccessMask = VK_ACCESS_INPUT_ATTACHMENT_READ_BIT |
                     VK_ACCESS_COLOR_ATTACHMENT_READ_BIT |
                     VK_ACCESS_COLOR_ATTACHMENT_WRITE_BIT |
                     VK_ACCESS_DEPTH_STENCIL_ATTACHMENT_READ_BIT |
                     VK_ACCESS_DEPTH_STENCIL_ATTACHMENT_WRITE_BIT,
    .dependencyFlags = 0
};
```
这很好理解，就是保证 subpass 能读到已经 available 的 attachments。类似地，如果最后一个使用该 attachment 的 subpass 到 `VK_SUBPASS_EXTERNAL` 之间没有 subpass denpendency，那么会隐式地插入一个
```
VkSubpassDependency implicitDependency = {
    .srcSubpass = lastSubpass, // Last subpass attachment is used in
    .dstSubpass = VK_SUBPASS_EXTERNAL,
    .srcStageMask = VK_PIPELINE_STAGE_ALL_COMMANDS_BIT,
    .dstStageMask = VK_PIPELINE_STAGE_NONE,
    .srcAccessMask = VK_ACCESS_COLOR_ATTACHMENT_WRITE_BIT |
                     VK_ACCESS_DEPTH_STENCIL_ATTACHMENT_WRITE_BIT,
    .dstAccessMask = 0,
    .dependencyFlags = 0
};
```
这就是保证之前对 attachments 的写变得 available

### Sync Guarantee in Queue Submission
从 [Host Write Ordering Guarantees](https://registry.khronos.org/vulkan/specs/latest/html/vkspec.html#synchronization-submission-host-writes) 以及 [Availability, Visibility, and Domain Operations](https://registry.khronos.org/vulkan/specs/latest/html/vkspec.html#memory-model-vulkan-availability-visibility) 可以看出，queue submission 保证了本次提交的，以及后续提交的所有 command 都可见之前 host domain available 的写

这意味着，大部分情况下，host write 除了 `vkFlushMappedMemoryRanges` 之外不需要其它 barrier 来进行同步了。不过 [Yet another blog explaining Vulkan synchronization](https://themaister.net/blog/2019/08/14/yet-another-blog-explaining-vulkan-synchronization/) 中提到了使用 event 在 host 和 gpu 同步时，可能出现 queue submission 之后的 host write 需要在 gpu 侧可见，这时不仅需要 `vkFlushMappedMemoryRanges`，还需要显式的 barrier 
### Memory Sync in Host and Device
#### Host Memory Flush
根据 [vkFlushMappedMemoryRanges](https://registry.khronos.org/vulkan/specs/latest/man/html/vkFlushMappedMemoryRanges.html) 和 [vkInvalidateMappedMemoryRanges](https://registry.khronos.org/vulkan/specs/latest/man/html/vkInvalidateMappedMemoryRanges.html) 的文档，我们也可以进一步解释它们的同步语义，前者是一个 *availability operations*，使得 host 上的写在 host domain 上 available。后者是一个 _visibility operations_，使得在 host domain 上 available 的写，对后续 host memory 上的读写可见
#### Sync Example
回过头来看 [Synchronization-Examples-transfer-dependencies](https://github.com/KhronosGroup/Vulkan-Docs/wiki/Synchronization-Examples#transfer-dependencies)

第一个例子 [Upload data from the CPU to a vertex buffer](https://github.com/KhronosGroup/Vulkan-Docs/wiki/Synchronization-Examples#upload-data-from-the-cpu-to-a-vertex-buffer)。要将 host 数据向 gpu 的传输，首先是创建一个 staging buffer，然后将 staging buffer map 到 host 上，host 正常向 staging buffer 填充数据。填充完毕后，需要使用 `vkFlushMappedMemoryRanges`（*availability operations*）。然后是两个 command，第一个 command 负责将 staging buffer copy 给 vertex buffer，第二个 command 是一个 barrier，使得对 vertex buffer 的写对后续的 vertex buffer 的读可见。**而 command 的提交保证了后续 device 的读写操作 可见 host domain 中 available 的写操作（从而第一个 copy command 能读到正确的数据）**

第二个例子 [Upload data from the CPU to an image sampled in a fragment shader](https://github.com/KhronosGroup/Vulkan-Docs/wiki/Synchronization-Examples#upload-data-from-the-cpu-to-an-image-sampled-in-a-fragment-shader) 和第一个例子类似，只是对于 image 来说，需要两个 barrier，第一个 barrier 负责将 image 的 layout 切换为 `VK_IMAGE_LAYOUT_TRANSFER_DST_OPTIMAL`，第二个 barrier 负责将 image 的 layout 切换为目标 layout，同时使得后续的 read 能看到之前 transfer write

第三个例子 [CPU read back of data written by a compute shader](https://github.com/KhronosGroup/Vulkan-Docs/wiki/Synchronization-Examples#cpu-read-back-of-data-written-by-a-compute-shader)，在 dispatch compute shader command 之后需要插入一个 barrier，其中 `VK_ACCESS_2_HOST_READ_BIT_KHR` 保证 compute shader 中对 memory 的写操作在 host domain 也变得 available，后续 `vkInvalidateMappedMemoryRanges` 再使得后续 host domain 上的读写操作可以看见之前 compute shader 中对 memory 的写操作

如果 device local 的 memory 也是 host visible 的，那么填充 device local 的 buffer 则不需要借助 staging buffer。但 optimal tiling 的 image 还是依赖 staging buffer，因为它的 memory layout 是未知的。但在 vulkan 1.4 中或者开启 `VK_EXT_host_image_copy` extension，则 host 侧可以通过 `vkTransitionImageLayout` 直接调整 image 的 layout，以及 `vkCopyMemoryToImage` 直接填充 image

**注1：`vkTransitionImageLayout` 和 `vkCopyMemoryToImage` 都不涉及 command，因此这两个函数都只涉及 cpu，这意味着用户需要保证在 cpu 访问该 image memory 时，gpu 不会读写该 image memory。另一方面，如果此前在 device 侧有写过该 image memory，也需要使用带 `VK_ACCESS_HOST_READ_BIT` 的 barrier 保证 host 侧能看到该写操作**

注2：`vkTransitionImageLayout` 和 `vkCopyMemoryToImage` 文档中都要求 **The memory of `image` is accessed by the host as if [coherent](https://registry.khronos.org/vulkan/specs/latest/html/vkspec.html#memory-coherent)**。再加上 queue submission 保证了 host 这边 available 的写都会变为 device 侧的 visible，**因此除了注 1 外，不再需要额外的同步操作了**
